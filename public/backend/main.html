Specification

Web Architecture

The Insieme web application is a presentation website for the Insieme Association, which is helping foreigners study the Italian Language and integrate easier into the Italian society.


In order to separate the business logic from the rendering and styling of the elements, the functionality is separated in 2 sections:

  1. The Presentation Layer
  2. The Application Layer

Because most of the data presented on the website is prone to changes or addition of new information, the Application Layer is communicating with the third component of the system:

  1. The Data Layer

This split allows changes to be independently tackled in the required component without the need of chaging the interface, just the implementation. In this way, components that communicate with the updated component do not need to be aware of any internal change.
This separation follows the 3 Tier Architecture, one of the most popular used in web development.

The application is respecting SOLID object oriented programming principles, so each component has a clear role, and it is easy to develop components individually. They interract with eachother through method calls and ocasionally promisses.

1. Presentation Layer

The Presentation Layer refers to the interface of the website that the users see and interract with. It consists of different components that communicate and interract with eachother. This part of the application is also reffered to as front-end.

Web Browser

Data is rendered and displayed for the user in a web browser. The web browser is receiving from the user different instructions, such as loading the website at the address “https://insieme-hya.herokuapp.com/”. The browser will make a GET HTTP(S) call to the server that hosts the website at the specified address, and receives a set of static data, usually HTML, CSS and Javascript. The browser interprets them and displays the result. Usually, the main file served when first accessing a website is index.html

INDEX.HTML

It is the main file that is rendered by the browser when the user accesses the website on the root address https://insieme-hya.herokuapp.com/. It contains the homepage and it acts like a gateway towards the other pages of the website, having call to action buttons towards other pages of the website.

Pages

The pages are HTML files that contain the template over which the web pages are build. They are including CSS files and JavaScript files from the assets folder, which help render the final page. Each page has a HTML template defined in the Pages folder, such as allcourses.html, event.html.

Assets

The assets are static documents that are included in HTML template in order to manage the style and the functionality of the page, like loading data and adding it in the DOM, or styling HTML elements. DOM (Document Object Model) is an object build by the browser when the web page is loaded.
The JavaScript files are taking care of building the functionality of the website and they make requests to the website’s REST API in order to get data from the database, and render it in the DOM.

2. Application Layer

The Application Layer refers to the business logic part of the website, where dynamic (changing) data is handled. It has different components and sub-layers that help with separation of concerns and independent development of componets. Users do not interract with this section directly, so this it is considered as being part of the back-end of the web application.
The role of this part of the website is to handle dynamic data, meaning data that is not stored in the code of the application, but in a database. The Application Layer provides the needed data to the front-end, which renders it with the help of the assets. In this way, all the functionality related to rendering and displaying data is managed by the front-end, whereas all the data handling is managed by the application layer.

INDEX.JS

This is the main file that is used to start and run the server that hosts the web application. In it there are settings for making Swagger REST API work, as well as configurations for the server (such as port, or static files path).

REST API: SWAGGER

The first component of the application layer is the REST Interface. A REST interface is a collections of resources that can be accessed (called) over HTTP protocol, so that different types of clients can communicate with the application layer. For example, we defined a rule on GET over the URL “https://insieme-hya.herokuapp.com/v1/courses”, so that the JavaScript files attached to the page allcourses.html can make a GET request to this URL, and expect a list of courses.
We used the Swagger library for defining those resources, but also for building a documentation (that can be accessed at: https://insieme-hya.herokuapp.com/backend/swaggerui). There one can find the collection of all supported REST API resources, with the possibility to test them.

CONTROLLER LAYER

The Controller Layer contains the components that are catching the requests that come from the front-end and distributes the appropriate commands to the service components. When the answer from the services comes back, the controller component is responding over HTTP to the front-end.
The application has a controller for every type of independent entity that we have in the database:

  1. Course
  2. Event
  3. Person
  4. Comment
SERVICE LAYER

The Service Layer contains the components that receive tasks (calls) from the controllers and they are the ones which implement all the logic needed for the dynamic features of the application. They manage and manipulate data that comes from the database before sending it back to the controller, as well as manipulating data that comes from the controller before saving it in the database.
The services communicate directly with the corresponding controller components, from where they receive calls, and with the DAO (Data Access Object) componens that are the ones which have access to the database. They call the DAO components when they need to save or get data.
The application has a service for every type of independent entity that we have in the database:

  1. CourseService
  2. EventService
  3. PersonService
  4. CommentService
DATA ACCESS LAYER

The Data Access Layer consist of components of type DAO (Data Access Object), which have direct access to the database. Only this layer is able to communicate with the database, so if some things change in the database, changes can be done in the corresponding DAO class without changing the rest of the application.
The DAO components receive calls from the service components, which request data or send over data to be saved in the database. The DAO components make sure the database has specific tables(otherwise it requests the tables to be built), that incoming data is mapped to the coresponding columns in the database, and that the frontend receives only the needed data.
The application has a service for every type of entity that we have in the database:

  1. CourseDAO
  2. EventDAO
  3. PersonDAO
  4. CommentDAO
  5. ResourceDAO - “Resource” is not an independent component, being strictly dependent on Course, but we decided to create a DAO component that handles the basic operations, instead of adding them all to CourseDAO.

3. Data Layer

The Data Layer refers to the storage of data in a database, assuring the dynamic features of the website.
In the database, each independent entity that has data to be stored has its own table:

  1. course
  2. event
  3. person
  4. comment
  5. resource

Besides the entities, there are 2 tables that are used for mapping 2 many to many relationships:
6. course_presentation - which maps the relationship between courses and events
7. course_volunteers - which maps the relationship between courses and volunteers (people)

Those tables are created by the DAO components, and basic SQL operations are performed only by the Data Layer.

API

REST compliance

Building this application that uses a REST API to mediate the communication between back-end and front-end, we wanted to respect as much as possible the REST Principles, not only because they are good practices, but also because they are widely known and any developer that may take over the work on Insieme website will be able to easily understand the functionality that the application layer provides.

  1. Cliend-Server - The app has a strong separation between the functionality executed on the client server, which is mainly rendering data and information, and the functionality executed on the server side which is data handling and manipulation.
  2. Stateless - There was no need for stateful or session dependent data to be kept on the server, therefore the app is treating any request as independent and complete
  3. Uniform interfaces - The API documentation is describing the resources what the app is using, as well as their models. Moreover, the requests follow a similar structure, so once you understand how to get data for one of them, you can follow the same pattern for the others. For example: /person/{personID} , /events/{eventID} , /course/{courseID} are all retrieving the requested resource with the specified ID.
  4. Layered system - The application has components that are split in layers by their role, and they can communicate only with the layers that are addiacent to them. This assures the separation of concerns and a better individual development of components.

Besides these principles, we also needed to decide how to map the requests that need data from 2 or more entities, especially for entities that have a relation in the database.
For this, we decided to use the following struncture for the url:
"/main-entity-name/{main-entity-id}/related-entity-name"
where main-entity-name and related-entity-name have a relationship in the database (usually one to many or many to many), and this resource will provide all the entries of type “related-entity-name” that are in a relation with the entry of type “main-entity-name” that has the id equal to main-entity-id/. For example, making a GET request on “/person/1/events” will retrieve all events that are in a relation to the person with id = 1.

OpenAPI Resource models

In order to ensure a good communication between back-end and front-end, we introduced models for the resources, that they have to respect.

Course

The Course resource represents information the services that the association is offering, which are Italian Language courses. It has the following properties:

  • id: integer - the unique identifier of a course
  • level: enum [‘Introductory’,‘Basic’,‘Intermediate’] - the level of the course
  • daysOfWeek: string - the days of the week in which the course is held; it has to respect the format ‘Monday;Tuesday;Sunday’ - a string with the day of the week separated by ‘;’
  • times: string - the times of the day in which the courses are held; it has to respect the format ‘16:00;17:00;18:00’ - a string with the hours separated by ‘;’
  • location: string - the address where the course is held
  • cerf_level: string - the CERF level of the course
  • volunteers: array of Person - an array of objects of type Person that represent the volunteers (people) that are involved in the course; the items will have the following properties:
    • id
    • name
    • photo
  • events: array of Event - an array of objects of type Event that represent the events that are recommended for this course; the items will have the following properties:
    • id
    • title
    • date
    • location
    • description
    • photos
Event

The Event resource represents information about an event that the association is organizing. It has the following properties

  • id: integer - the unique identifier of an event
  • title: string - the title of the event
  • date: string (format: date-time) - date in which the event happens
  • description: string - description of the event
  • photos: string - urls to the photos that the event is representing; separated by “;”
  • coordinator: Person - an object that represent the person which is a contact for an event; the object will have the following properties:
    • id
    • name
    • photo
  • courses: array of Course - an array of objects of type Course that represent the courses for which this event is recommended; the items will have the following properties:
    • id
    • level
    • cerf_level
Person

The Person resource represents information about a volunteer that is involved in the association. It has the following properties

  • id: integer - the unique identifier of a person
  • name: string - the name of the person
  • role: enum [‘teacher’,‘coordinator’] - the role of the person
  • description: string - description of the person
  • photo: string - urls to the photos of the person; separated by “;”
  • job: string - job of the person
  • city: string - city where the person is from
  • quote: string - a quote of the person
  • courses: array of Course - an array of objects of type Course that represent the courses in which this person is involved; the items will have the following properties:
    • id
    • level
    • image
    • description
  • comments: array of Comment - an array of objects of type Comment that represent the student comments (reviews) about this person; the items will have the following properties:
    • id
    • text
    • date
    • student_name
    • photo
Resource

The Resource resource represents information about a study material that is recommended for a specific course. It has the following properties:

  • id: integer - the unique identifier of a resource
  • title: string - the title of the resource
  • cerf_level: string - the CERF level of the resource, for example: A1,A2
  • url: string - url to the resource
  • course_id: integer - the unique identifier of the course for which this resource is recommended
Comment

The Comment resource represents a review or a testimonial given by students that have participated into the association’s courses and events. They can be reviews of the association, or comments about a specific volunteer (teacher). It has the following properties:

  • id: integer - the unique identifier of a comment
  • text: string - the text containing the opinion (comment)
  • student_name: string - the name of the student which made the comment
  • photo: string - url to the photo of the student
  • date: string (format: date-time) - date in which the comment was made
  • person_id: integer - the unique identifier of the person about who this comment is; if it is null, then the comment is not about a person, but about the association on the whole.

Data model

In the OpenAPI specifications, there are 5 data models for the main entities of our database: Person model, Course model, Event model, Resource model and Comment model.

The Resource model and the Comment model have the same attributes as in the ER model. On the other hand, additional data was added to the other three models in order to fulfill the relationships between the entities. A Person, for example, will have associated the Course in which he/she is involved, the coordinated Events and the Comments (reviews) that the students have made about him/her, with some of their properties as well.

In the same way, a request to get an Event will carry the Person coordinating that Event, and the Course that is associated to the Event; and finally, a Course will also give information about each Person teaching it and the Events that are recommended to students following that Course.

Implementation

Tools used

JavaScript

JavaScript is the main programming language used for web development, that is able to bring functionality upon otherwise static HTML files. The language is known and interpreted by all browsers, and a lot of frameworks and libraries based on it are created each year.

Node.JS

Node.JS is a runtime environment which can run JavaScript outside of a browser. It is widely used for back-end development in the web industry and it facilitates the creation of servers that are able to handle HTTP requests and communicate to other services or databases.

Swagger with OpenAPI Specifications

Swagger is a framework that is used for easily creating documentation and managing REST APIs. It is based on OpenAPI Specifications, that are rules for configuring REST APIs in a JSON or YAML document. Swagger frameworn offers data validation, routing the request to appropriate controls, an interface for the documentation in which requests can be tested out, and many more.

KNEX.JS

Knex.js is a query-builder library for JavaScript in which you use different methods in order to connect and perform operations over databases. The Insieme web app uses this library for creating tables, saving and retrieving data from the database.

PostgreSQL

PostgreSQL is the most advanced open source relational database, which is better suited for complex application that MySQL, another popular choice for web applications.

Heroku

Heroku is a Platform as a Service (PaaS), that allows developers to easily deploy and test their application in the cloud, without prior knowledge of server management and configuration. It comes with a CLI (Command-Line Interface) that makes it very easy to deploy the latest version of a website online. If the website needs a database, Heroku offers the possibility to add a PostgreSLQ database as an addon, that you can connect to the web app.

Discussion

How did you make sure your web application adheres to the provided OpenAPI specification?
In order to respect OpenAPI specifications, we built the application starting from them. This approach let us define the resources that the application offers, as well as the models, and we followed the recommended basic configuration that they should have. We ensured we have comprehensive examples for the models, and when we reached a good starting point, we generated the Node.JS application using Swagger Generate Server tool. In this way, we started developing from a point that was in sync with the OpenAPI specifications.
The changes or additions to the specification were done in the same time with the functionality, so that we continously updated the document when things changed.
Another important tool is the Swagger Validator which makes sure that the request values respect the data types and formats defined in the specifications file.
Which method did you use to test all APIs endpoints against the expected response?
We used the Swagger UI interface to test every endpoint and we solved the ocasional bugs when we noticed them. Alternatively, we used also Postman for making API request with testing purposes.
Why do you think your web application adheres to common practices to partition a REST-based web application (static assets vs. application data)?
We used the Swagger UI interface to test every endpoint and we solved the ocasional bugs The Insieme web app has a clear separation of static assets, such as HTML, CSS and JavaScript files that build the user interface (front-end), from the business logic JavaScript files that only handle operations with data. The front-end, static files are stored in the folder “public”, which is configured in the application server to serve its files as static. All the other files constitute the server functionality and they are only handling data operations through the Swagger routing and the controllers, which decide the mapping of the requests to actions.
Which technology did you use (relational or a no-SQL database) for managing the data model?
We used a relational Database because of the relatively big number of relations that were needed. Every entity is part of at least 1 relationship, with 2 relationships being many-to-many. In a NoSQL database, those would have been complex to map and mantain.
By using a relational database, we eliminated the need of keeping duplicate data.

Analysis of existing API

123FormBuilder API

Link: https://developer.123formbuilder.com/api-v2/
123FormBuilder is a platform that allows users to create forms and manage their submissions. They have an API documentation that developers use to integrate forms within other applications.
We have taken inspiration from 123FB API for implementing queries that were dependent on a relationship.
For example:

GET
/forms/{form_id}/submissions
retrieves all the submission that have been registered on a form with id = form_id.
We used the same approach for getting or saving a new resource for a specific course:
GET
/courses/{courseID}/resources
POST
/courses/{courseID}/resources

Eventbrite API

Link: https://www.eventbrite.com/platform/api
Eventbrite is a platform for creating and promoting events. We read their API documentation and checked out how they implemented the listing of events. In order to get the events, you have to choose between listing them:

  • by venue:
    GET
    /venues/{venueID}/events/
  • by organizer:
    GET
    /organizations/{organizationID}/events/
  • by series:
    GET
    /series/{seriesID}/events/

We used the same resource format on the relationship between people and events, allowing us to retrieve all the events for which a person is a contact:

GET
/person/{id}/events

Meetup.com API

Link: https://www.meetup.com/meetup_api/docs/
Meetup.com is a platform that facilitates community gatherings. You can create a community and have recurring events that are happening for its members.
When you create a new event, you can set this parameter:

event_hosts - String containing up to 5 comma-separated valid member ids for members who will be hosts of the event.
We used the same tactic where we needed to have a list of primitive values, such as in:

  • daysOfWeek in Course model: ‘Monday;Tuesday;Sunday’
  • times in Course model: ‘16:00;17:00;18:00’
  • photos in Event and Person models: ‘url1;url2’
